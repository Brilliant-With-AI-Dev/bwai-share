Email architecture for BWAI products

We need to send email from the BWAI product surface — starting with the Mirror Insights Snapshot and Mirror Report. Two viable architectures. This doc lays out both, plus the recommendation and the runner-up.

2026-05-18 Owner: Matt Audience: BWAI team Decision deadline: TBD

The context

What we'd send first

Mirror Insights Snapshot (Stage 3) and Mirror Report (Stage 4) — the consulting deliverables, not notifications.

Volume today

~0–10 sends/week. Grows with paid engagements.

Personalization

High. Body is the user's own insights — needs care, not a generic "your report is ready."

Sender identity

Always from BWAI (e.g. hello@brilliantwith.ai). Never from Matt. This is a product, not a founder.

Other email types coming

Auth/verification (Stytch backup), ops alerts, future stakeholder invites at scale (Discovery).

Current state in product

/mirror-next ships a mailto: link that opens the user's own email client with a truncated copy. Brittle and off-brand. Replace.

The recommendation at a glance

🥇 Recommend

Resend + AI-personalized body

One pipe for every email. Scales day one.

Programmatic transactional infra. Templates leave room for an AI-drafted body so deliverables still feel personal. Optional review-queue UI for high-stakes sends. Covers auth and ops with the same setup.

Use what we have. Defer Resend until needed.

Agent drafts each Insights/Report email into a shared BWAI Drafts folder. A human reviews and sends from a Workspace inbox. Zero new infra, free HITL — but bottlenecks on review and doesn't cover transactional email.

The two options in detail

🥇 Resend + AI-personalized body

Standard transactional infrastructure. Every email — deliverable, auth, ops — flows through one Resend account on a dedicated sending subdomain. Each deliverable template includes a slot for an AI-generated body, so the email reads like a thoughtful note instead of a generic notification. A lightweight review queue can gate high-stakes sends if we want HITL.

Sender: hello@mail.brilliantwith.ai (subdomain isolates transactional reputation from the marketing root). From-name: Brilliant With AI. Reply-To: a monitored shared inbox.

Setup cost: ~½ day. DNS records (SPF / DKIM / DMARC), one API key per environment, React Email templates for each deliverable, send endpoint in the Next.js proxy.

Run cost: Free up to 3,000 emails/month, then ~$20/mo.

Pros

  • One architecture for every email type — deliverable, auth, ops
  • Scales from 10/week to 10k/week with no rework
  • Instant, predictable delivery — no human bottleneck
  • AI-generated body preserves personalization quality
  • Bounce, complaint, and open tracking built in
  • Easy to add a review queue later if HITL becomes desired
  • Works the same in dev / staging / prod

Cons

  • DNS setup (~hour) and ongoing reputation hygiene
  • React Email templates to design + maintain
  • HITL is opt-in, not automatic — easier to ship something we'd want to review
  • Resend lock-in (mitigated: it's just an API; portable)
  • Replies don't land in a real inbox unless we wire Reply-To

🥈 Workspace + agent skill for deliverables

Use Google Workspace as the send surface for consulting deliverables. An agent skill ingests the Insights Snapshot or Mirror Report, drafts a personalized email body, and saves it as a draft in a shared BWAI Workspace inbox. A human (any BWAI team member) reviews the draft and sends from Gmail. Transactional email (auth, ops) still requires a provider — defer that decision until it's needed.

Sender: hello@brilliantwith.ai — a shared Workspace inbox, not a personal account. Replies land in a real inbox and thread normally.

Setup cost: ~½ day for the agent skill + Workspace MCP wiring. Zero infra. No DNS work.

Run cost: $0 incremental (Workspace already paid).

Pros

  • Free HITL by default — Drafts is the review UI
  • Highest-quality output — LLM tailors body per recipient
  • No DNS, no templates, no reputation management
  • Replies thread into a real inbox
  • Validates whether automated send is even desired before building it

Cons

  • Doesn't cover auth, ops, or any non-deliverable email — needs a second system
  • Human review is a bottleneck — fine at 10/week, broken at 100/week
  • No automation path — can't auto-send on report-ready
  • Workspace API quirks (OAuth refresh, rate limits, scope creep)
  • Multiple humans on the review rota risk inconsistency
  • Migration cost when we eventually flip to Resend — templates have to be rebuilt from agent prompts

Reference: pure transactional (no AI personalization)

For comparison only. Resend with static React Email templates, no AI-drafted body. Same infra as 🥇 but without the personalization layer.

Why we'd consider it: Simpler, no LLM cost or latency in the send path. Plenty good for auth and ops.

Why we wouldn't: Sterile for deliverables. The Insights Snapshot is the product; sending it under "Your report is ready" undersells what we just produced.

Verdict: Use this template style for transactional (auth, ops) under 🥇, and use the AI-body variant for deliverables.

Open questions for the team

  • Is "always from BWAI, never from a person" the right brand posture?

    Recommended yes. Founder-from undermines the product narrative.

  • For deliverables, do we want HITL on every send, or only for paying clients?

    If always: 🥈 has a real edge. If only sometimes: 🥇 with optional review queue wins.

  • Who owns the shared BWAI inbox if we go 🥈?

    Who reviews drafts, who covers when they're AFK, what's the SLA on send.

  • Are we OK shipping an AI-drafted body without review for low-stakes sends?

    If yes, 🥇 is straightforward. If no, the review queue work goes from "later" to "now."

  • When do we expect the first non-deliverable email need (auth, ops, billing)?

    If < 90 days, picking 🥈 means doing Resend work soon anyway — argues for 🥇 now.

Proposed next step

Pick 🥇 unless the team has a strong opinion on HITL-by-default for deliverables. Spike: stand up Resend on the staging subdomain, ship one AI-bodied template for the Mirror Insights Snapshot, replace the mailto: in /mirror-next. Two days end-to-end including DNS propagation.

The 🥈 path is reversible — if we pick 🥇 and decide later we wanted HITL, add the review queue. If we pick 🥈 first, we still have to build 🥇 anyway for the first auth or ops email.
