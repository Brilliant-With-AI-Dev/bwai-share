How should BWAI send email?

Mirror produces a personal report for each user. Right now there's no good way to email it. Before we wire anything up, we need to choose what sending looks like. Two real options.

2026-05-18 Owner: Matt Audience: BWAI team Read time: ~5 min

The question

When Mirror produces a report, how does it reach the user's inbox?

Today: a clunky button that opens the user's own email program with half the report pasted in. Not a real send. We need to replace this, and we need to decide the shape — not just pick a vendor.

What we'd be sending

The emails

Mirror Insights Snapshot and Mirror Report. The consulting deliverables themselves, not "your report is ready" notifications.

How many today

Maybe 5–10 per week while Mirror is being demoed. Will grow with paid engagements.

How personal they need to be

Very. The body is the user's own insights. A generic template would undersell what we just produced.

Who they're from

Always from BWAI — never a personal address. Mirror is a product, not a founder side project.

Other emails coming soon

Sign-in confirmations, system alerts, maybe stakeholder invites for the next product. We need a plan that doesn't get reworked every time.

What's there today

Nothing. No email goes out automatically. The current button is cosmetic and the experience is bad.

The recommendation

🥇 Recommend

AI writes it. The product sends it.

No human in the middle. Instant.

When the user's report finishes, the product writes a short, personal email about it and sends it straight to them. Same machinery handles sign-ins, alerts, and anything else we need to email. If we ever want a teammate to glance at a draft before it goes, we can add that later.

AI drafts it. A teammate sends it.

Every email gets a human glance first.

When the user's report finishes, the product writes a draft and drops it into a shared BWAI inbox. A teammate reads it and clicks send. Slower, friendlier, and we catch anything weird before it leaves. Doesn't help with sign-ins or alerts — those need something else.

The two options in detail

🥇 AI writes it. The product sends it.

When the Mirror report finishes, the product writes a short, warm, personal email about it — referencing what the user actually said — and sends it instantly to their inbox. The same sending pipe handles every other email type we'll ever need (sign-ins, alerts, future products).

What the user experiences

Finishes the Mirror interview

Sees the insights on screen

Email lands in their inbox within seconds, written for them specifically, from Brilliant With AI

They can reply; replies go to a shared BWAI inbox

What's good

  • One system covers every email we'll ever send
  • Instant. No waiting on a person to be at their desk
  • Each email is freshly written for the recipient
  • Works the same at 10 sends a week or 10,000
  • Easy to add a "teammate reviews first" step later if we want it

What's hard

  • Nobody on the team sees the email before the user does
  • If the AI writes something off, the user sees it before we do
  • Some one-time setup (~half a day of engineering)
  • Small monthly cost once volume grows (~$20/month at scale)
How we'd build it

Use Resend (a transactional email service) on a dedicated sending subdomain, e.g. mail.brilliantwith.ai. The Mirror app generates the email body using the same model that produced the report, then hands it to Resend to deliver. Reply-to points to a real shared inbox so replies thread normally.

Setup: about a half-day. DNS records for the subdomain, one account, templates for each email type, a small piece of code that calls the send API when a report finishes.

Cost: free up to 3,000 sends per month, ~$20/month above that.

Reversible: if we later want a teammate-review step, we add a queue and a simple approval screen. The send pipe doesn't change.

🥈 AI drafts it. A teammate sends it.

When the Mirror report finishes, the product writes a draft and drops it into a shared BWAI inbox. A teammate opens the draft, reads it, tweaks anything that looks off, and clicks send. Slower and more considered. Doesn't cover sign-ins, alerts, or anything else that needs to go out without a human in the loop.

What's good

  • Every email gets a human glance before going out
  • Catches anything weird the AI writes
  • Highest possible quality per email
  • Uses tools we already have and pay for
  • Replies land directly in the same inbox

What's hard

  • Someone on the team has to be on the rota to review drafts
  • Breaks down past ~30 sends per week or when reviewers are AFK
  • Can't send instantly — there's always a wait
  • Doesn't cover sign-ins, alerts, or anything automated — we'd still need a second system for those
  • If we later switch to option 🥇, the email templates have to be rebuilt
How we'd build it

Use the existing Google Workspace account. A small piece of code in the Mirror app generates the draft email and pushes it into a shared inbox using Workspace's drafts feature. The teammate opens Gmail, reviews, hits send.

Setup: about a half-day. Auth into Workspace, a script that creates drafts, a shared inbox to receive them.

Cost: nothing new. Workspace is already paid for.

Limitation: this is only good for Mirror's consulting deliverables. Sign-in confirmations and system alerts can't go through this — they'd need option 🥇 anyway, eventually.

Things we need the team to weigh in on

  • Is "always from BWAI, never from a person" the right brand stance?

    We think yes. Sending as a person makes Mirror look like a side-project, not a product.

  • For the Mirror report itself, do we trust an AI-written email to go out without a human reading it first?

    This is the single biggest fork. Yes → 🥇 wins easily. No → 🥈 wins, and we need a reviewer on the rota.

  • If we go 🥈, who owns the shared inbox? Who covers when they're off?

    Someone has to read drafts daily, including weekends if reports come in.

  • When do we expect to need the first non-Mirror email? (Sign-in confirmation, system alert, something else.)

    If within ~3 months, picking 🥈 first means doing 🥇 work soon anyway. Argues for skipping 🥈.

If the team agrees

Build 🥇 first. Ship one email — the Mirror Insights Snapshot — so we can see real sends and judge whether the AI-written body actually feels personal enough. About two days of engineering end to end. Replace today's broken button.

If, after seeing real sends, the team wants a teammate-review step, we add one. The cost of changing our mind later is small. The opposite is not true — starting with 🥈 means rebuilding when we eventually need 🥇 anyway.
