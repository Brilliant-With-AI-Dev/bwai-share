What is this

A short guide to our "feature flags" — the on/off switches that let us hide a finished feature from customers until we're ready.

Who it's for

Anyone on the business or marketing team who needs to review or test a new feature before it goes live.

What you get

Three clear ways to see a not-yet-live feature, with step-by-step directions and pictures of each screen.

How it used to work

  • A feature was either live for everyone or not built at all.
  • Testing meant either guessing or risking customers seeing half-done work.

How it works now

  • Finished code can sit on the live site with its switch OFF.
  • You can turn it ON privately, test it, then launch on purpose.

Why it matters to you

  • You catch problems before any customer does.
  • If a launch goes wrong, we undo it in seconds, not hours.

The one idea: a switch on every feature

Every new feature we build has an on/off switch attached to it. Engineers call it a feature flag; you can just think of it as a light switch.

The code for the feature ships to our website like everything else, but the switch starts in the OFF position, so customers don't see it yet. We can turn it ON privately to test, fix anything that's wrong, and only flip it ON for everyone once we're confident.

The curtain analogy. Picture a stage with the curtain closed. The set is fully built and the actors are in position, but the audience sees nothing until we choose to raise the curtain. A feature flag is that curtain.

What are you trying to do?

Path 1 — Preview link Recommended, no setup

While an engineer is building a feature, every version of their work automatically gets its own private preview website. It's a complete, working copy of our product where the new feature is already switched ON.

  1. Ask the engineer for the preview link. It looks like a normal web address ending in .vercel.app.
  2. Open it in your browser and use the product normally. The new feature is usually already on.
  3. Test it, click around, and send your feedback back to the engineer.
If you hit a login wall or don't see the feature: tell the engineer. They'll either give you access or confirm the right link. Each feature decides for itself whether it's on in previews, so occasionally one is off by default.

This is the path to use 9 times out of 10: fastest, nothing to install, zero risk. It uses a test login, not your real customer account — the engineer will tell you how to sign in.

Path 2 — Test a feature on the real live site

Sometimes you need to see a feature on the real production website (brilliant-apps.vercel.app) — for example, to test it with real data. You can switch a hidden feature ON for your own browser only. Customers keep seeing the product exactly as it is now.

Each time you want to test

  1. Open the live site, brilliant-apps.vercel.app. setup done
  2. Find the Vercel Toolbar — a small dark bar at the bottom of the page — and click the flags icon to open the Flags Explorer.
    See the picture below for exactly which icon.
  3. Find the feature by its name. An engineer will tell you the exact name, for example release.b2c-consumer-surface.
  4. Set it to "On."
  5. Refresh the page. You now see the feature; everyone else still sees the normal site.
It only affects you. The switch lives in your browser, it's temporary, and it doesn't follow you to another computer. To return to the normal customer view, set it back to Off or just close the browser.

Path 3 — Turn it on for everyone (the launch)

When the team agrees a feature is ready, an engineer flips its switch to ON for all customers. This happens instantly, with no new version of the site to build and wait for.

You don't do this yourself — it's the actual release decision — but two things are worth knowing:

  • It's instant in both directions. If something looks wrong right after launch, the engineer flips the switch back to OFF in seconds and customers return to the previous experience. No drawn-out rollback.
  • "Ready to launch" is a business call as much as a technical one. Your sign-off on a feature (from Path 1 or 2) is usually what tells the team it's safe to raise the curtain.

A worked example

Say we've built new customer sign-up and login pages but aren't ready to open them to the public. They live behind a switch called release.b2c-consumer-surface, which is OFF for customers. Right now, anyone who tries to reach those pages is quietly sent back to the normal entry screen.

  1. Marketing checks the sign-up copy and flow. They ask the engineer for the preview link (Path 1), walk through sign-up as a new customer would, spot two wording changes, and send them over. No customer ever saw the work-in-progress.
  2. A final check on real production. A teammate who's set up on Vercel (Path 2) flips the feature ON in their own browser on the live site, confirms it behaves correctly with real data, then switches it back off.
  3. Launch day. With sign-off in hand, an engineer turns the switch ON for everyone (Path 3). Sign-up goes live for all customers at once. If a problem appears, they flip it back off immediately.

Common questions

If I turn a feature on for myself, will customers see it?
No. Paths 1 and 2 only change your own view. Customers only see a feature after Path 3, the launch.
Can I break the live product by testing?
No. Preview links are a separate copy, and the per-browser switch only changes what you see.
A feature I was told about isn't showing up. What now?
Tell the engineer the feature's name and which path you tried. Usually it's a wrong link, a switch that's off by default in previews, or a setup step missing on your Vercel account.
Why don't we just push features straight to customers?
Because "the code is on the website" and "customers can use the feature" are now two separate decisions. That lets us test safely in the real environment first, launch on a deliberate schedule, and undo instantly if needed.
Is this the same as LaunchDarkly or other flag tools I've heard of?
Same idea, simpler setup. We use Vercel's built-in flag tooling rather than a separate paid platform. For your purposes, the experience is exactly what's described above.
