Section 1

What's happening today (the Rube Goldberg)

To run a mirror-next interview today, the bapps browser builds the system prompt, sends it as a fake user message to a "transcriber" step that copies it verbatim into a context asset, then a second step interpolates that asset back as its system prompt. Report and closing triggers are smuggled in <hidden>...</hidden>-wrapped user messages. Form data does the same.

It works. But it's slow, brittle, and any user with DevTools can read the system prompts because they cross the wire from the browser.

Problem node: prompt leak, latency, or transport hack Plain hop Fidelity loop (transcribed text checked against intended prompt)

Three problems baked in

  • Detour through transcriber — adds a full round-trip (taskCreate + chatStream + fidelity check, ~3–6s on the "preparing" screen) before the user can even see the AI's first question. Drifts ~1–3% on length even when "fidelity ok".
  • <hidden> hack — system instructions ride inside user-role messages; the AI has been told to obey them anyway. Same trick for report triggering, closing wrappers, and form-data context. Sanitisation is a moving target (see form-input.ts).
  • Browser DevTools sees everything — the full interview prompt, the report-mode flip, the closing wrapper, and verbatim form data all cross the wire in POST /api/... request bodies. Any user who opens Network can copy the whole IP.

Section 2 · End shape v1

Parity-Plus — same product, clean internals

TWAI platform 1.8.0 added a delegated machine-to-machine (M2M) surface: bapps's server can mint a token, call the platform on a user's behalf, and set the system prompt directly on a step via promptOverrides.system. It also added role: system / role: developer chat messages that can be appended mid-conversation (with or without an LLM response).

End shape v1 uses those primitives to deliver identical user-visible behavior as mirror-next today, with three structural wins: no transcriber detour, no <hidden>, no prompts crossing the browser. The browser sends real user messages; bapps's server holds the system instructions.

Why this is the right v1

Identical product behavior. Three eliminations (transcriber, <hidden>, browser-side prompts). One new route (/api/m2m-chat), one new auth helper (server-side sessions.authenticate), one new token cache. No new platform asks beyond provisioning the M2M client on usertests' Stytch project — the capabilities are already shipped. Risk is bounded; the cutover is what the migration plan calls "synchronized" (the chat-body shape change must merge the day usertests goes ≥1.8.0).

Section 3 · End shape v2

Next-Gen — first-principles, agentic

If you start from the business outcome — "interview a stakeholder in a way that feels human, drives complete coverage of the topic guide, and produces sharp, structured insights with minimal effort" — the shape stops looking like a single chat with a system prompt. It looks like a small team of cooperating agents, each with a narrow job, sharing typed state in context assets.

This section proposes seven moves, scores each against what the platform supports today, and is honest about the gaps. A few are hard nos — those are the exec signal for what to ask the platform team to build next.

The seven moves

1. Multi-agent decomposition: Interviewer + Quality Observer + Synthesis

Today one prompt does three jobs: drive rapport, track coverage, and eventually write the report. Split them. Interviewer talks to the user (warm, curious, follow-ups). Quality Observer watches the transcript in the background, scoring coverage of each interview-guide bucket; when a bucket looks thin, it appends a role: developer nudge to the Interviewer's history ("the user hasn't grounded their AI friction in a concrete example yet — ask for one"). Synthesis drafts insight fragments as the interview progresses, not at the end — so when the user hits "wrap it up", the report is already 80% written and just needs final assembly.

2. Structured-output extraction in-flight

The platform's chat response already returns {visible_message, data} with additionalProperties: true. Use the data field every turn to emit typed fragments — {goals: ["..."], pains: [{description, evidence}], decision_criteria: [...], confidence: 0.7}. Bapps accumulates these in a typed insight store; the report-generation step doesn't re-extract anything, it just composes what's already there.

3. Append-without-submit for rolling interview state

Replace the {# meta:progress=N,ready=...,qn=... #} suffix-parsing hack with periodic role: developer messages the user never sees, summarizing where the conversation is ("covered: goals, current_workflow. uncovered: success_metrics, blockers. emotional_register: pragmatic. proceed to: blockers"). The model references these naturally in the next turn. No regex, no leaky tags.

4. Context-asset memory across sessions

Persist long-form discoveries (the user's stated 12-month goals, the named friction with their current AI tool, their decision-criteria for a vendor) as named context assets at scope: user via the new /api/context-assets M2M endpoints. A follow-up interview six months later can read them back, ask "are these still the goals?", and pick up where we left off. Today everything is lost the moment the task finishes.

5. Mid-interview pivot: Quality Observer can re-steer the Interviewer

If the conversation drifts (user is venting, or has flipped from "AI is fascinating" to "AI is a threat"), the Quality Observer should be able to rewrite the Interviewer's instructions: e.g. "switch from open-curiosity stance to validating + grounding stance". Today promptOverrides.system is the only way to set a full system prompt, and it's accepted only on the first request of the session — so the alternative is to append a strong role: system mid-conversation. Workable, but stylistically narrower than "swap personas".

6. Tool-call hooks: AI requests file upload, calendar, schedule follow-up

A great interviewer can say "can you share the deck you mentioned?" and the UI surfaces a file picker; or "want to schedule the next session?" and a calendar widget appears. The platform's chat endpoint doesn't expose tool/function-calling. Today this would be faked client-side by parsing trigger strings out of the assistant message — a regression to the <hidden> pattern we're trying to kill.

7. Voice-first cues: pacing, interruption, prosodic "I'm thinking"

The interview is the product. A real interviewer pauses, mhms, leaves space. The platform mints a Deepgram token (/api/voice/token), so transcription/TTS is unblocked, but the conversational pacing — interruption handling, mid-utterance "let me think", barge-in — is all client-side and outside the prompt-flow shape.

Where the platform needs to grow (the v2 ask list)

  • Tool calling — first-class support for tools input on taskStepChat + tool_calls output channel + role: tool response messages. Without this, every "AI asks for a file" interaction is a regex on a string. This is the single biggest agentic-UX gap.
  • Mid-session prompt-override — relax the "only first request" constraint on promptOverrides.system. Today's append-only role: system doesn't replace the layer-4 template, only stacks on top of it. The Quality Observer can nudge but can't pivot persona.
  • Per-step JSON-schema enforcement — let bapps attach a Zod-equivalent schema to the step; platform 400s if data doesn't conform. Today the schema lives in our client and a drift goes silent.
  • Multi-agent / sub-task primitive — partial workaround exists (separate task steps, append-without-submit), but a real "spawn observer on this transcript" concept would simplify the orchestration enormously and unlock concurrent observer + interviewer.
  • Streaming partial structured output — verify that data partials stream as the assistant generates, not only when the message is complete. The exec value is the UI being able to highlight extracted insights live.

    For v1 (Parity-Plus)

    • Confirm usertests is on TWAI 1.8.0+ (currently inferred ~1.7.0; the 1.8.0 chat-body shape change won't work until usertests upgrades). This is the trigger event for the P0 cutover.
    • Register a brilliant-apps M2M client in usertests' live Stytch project and add its client ID to usertests' STYTCH_M2M_ALLOWED_CLIENT_IDS. Give Matt the four env vars (STYTCH_M2M_CLIENT_ID, STYTCH_M2M_SECRET, STYTCH_PROJECT_ID, STYTCH_SECRET) for the development-brilliant-apps Vercel project.
    • Confirm the Stytch M2M token endpoint host: https://api.stytch.com/v1/public/<project>/oauth2/token for the usertests live project; client_credentials is the right grant; m2m:inject:task-step-chat scope only.
    • Decide whether the mirror-next task template carries a separate report step: drives whether report-trigger uses promptOverrides.system on a new step or a role: system append on the existing one.
    • Confirm mid-step role: system ordering: an appended system message must land before the next user turn in history to actually steer that turn (migration plan open-question 7).

    For v2 (Next-Gen)

    Everything above, plus net-new platform asks — ranked by exec value:

    • Tool / function calling on taskStepChat — add a tools input field, a tool_calls output channel in the response, and accept role: tool as a result-of-tool-call message. Without this, no in-conversation file upload, calendar, or follow-up scheduling without a regex-on-string regression.
    • Relax promptOverrides timing to allow mid-session overrides (today rejected after first request). Enables Quality Observer to pivot the Interviewer's persona, not just nudge it.
    • Per-step JSON-schema enforcement — let bapps attach a schema to a step; platform rejects responses whose data doesn't conform. Today schema drift is silent.
    • Streaming partial structured output — confirm (or implement) that the data object streams partials during generation, not only at end-of-message, so the live insights panel updates as the AI talks.
    • Multi-agent primitive (nice-to-have) — a way to spawn an observer / side-agent on an existing task-step's transcript without bapps fanning out its own taskStep calls. Not a blocker; would simplify orchestration.
    • Voice pacing / interruption helpers are out of scope of prompt-flow shape — client-side problem with Deepgram + a state machine. No platform ask.

    Honest read

    v1 is ~all bapps work plus a provisioning ask. v2 is partly bapps orchestration we can do today (multi-agent via fan-out, structured output, context-asset memory, append-without-submit state) and partly platform features that don't exist yet (tool-calling, mid-session prompt-override, schema enforcement). A pragmatic path: ship v1 first, then build a "v1.5" that adopts the v2 capabilities the platform already supports (especially structured-output extraction and context-asset memory), and queue the platform asks for genuine agentic behavior.
