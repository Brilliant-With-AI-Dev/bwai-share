Mirror — entry routing, the proposed design

The complete walkthrough for the entry-routing refactor (branch feat/mirror-entry-routing), to bless before any code lands. Today, opening /mirror can flicker /mirror → /mirror/report → /mirror/start for a brand-new user, who lands for a moment on a URL named “report” that has no report. This change makes /mirror read the signed-in user's run state once, server-side, and redirect once to the single destination that is true for them. Everything is behind the release.mirror-return flag (on in preview/dev, off in production).

What changes (four edits, all flag-dark)

  • New resolver — lib/mirror/entry-routing.ts. One server-only function decides the destination from run state, reusing the exact read path /api/mirror/state already uses.
  • /mirror becomes an honest server redirect. It calls the resolver (with a timeout + safe fallback) and redirects once — no guess, no double-bounce.
  • New surface — /mirror/resume. “Pick up where you left off?” moves off the report page into its own route, so each surface owns one job.
  • /mirror/report slims to finished-report-only. It drops the resume gate; it renders the completed Mirror (and its own retry state on a platform hiccup).

1 · The decision table

The whole refactor in one place: a first-match-wins ladder. The entry router walks it top to bottom and stops at the first row that applies. Rows 1–5 are the cheap or defensive exits; rows 6–8 are the real three-way decision after run state is read. “Server can see it?” flags whether the choice is made server-side (at /mirror) or has to be finished client-side on the destination surface, because the browser session envelope is invisible to the server.

# Condition (first match wins) Decided Destination Why
1 release.mirror-return is OFF server /mirror/start Dark launch. No cookie read, no platform call — byte-for-byte identical to production today. The whole mechanism is invisible until the flag flips.
2 Flag on, but no Stytch session cookie server /mirror/start Anonymous visitor. The login wall lives on /start (the layout soft-gate), so entry short-circuits before any platform work — instant, exactly like today.
3 Flag on, cookie present, but PLATFORM_URL unset server /mirror/start Misconfiguration. Can't read run state with no platform, so default to the intro — recoverable, never a crash.
4 Session verify returns NO_SESSION (cookie present but no valid session) server /mirror/start The cookie is stale/garbage. Treat as logged-out and send to the intro; the gate on /start handles re-auth.
5 Resolver throws or times out: a transient session-verify error, M2M-token mint failure, asset-read failure, or the ~3500 ms entry timeout server /mirror/resume (fallback) We can't tell what the user has, so degrade to /resume, which re-probes and, if still failing, forwards to /report's explicit retry state. A returning user is never wrongly dumped on /start during an outage.
6 An incomplete run snapshot exists (mirror/state/<id> not yet in the manifest) server /mirror/resume A conversation to pick up exists — the highest-value honest destination, so it wins over a finished run. This is what the resume offer is for.
7 No incomplete run, but a finished run is recorded in mirror/index server /mirror/report Show the saved Mirror. A completed run is never offered for resume.
8 Authenticated, but no runs at all (no snapshot, empty manifest) server /mirror/start The brand-new-user path. One redirect to the intro — the wart's flicker is gone.
+ This browser tab is mid-interview (a live session envelope is present) client /mirror/interview The server can't see sessionStorage, so this is enforced on arrival: /resume and /report share a one-line guard that continues a live tab into the interview before doing anything else. See Use Case 4.
How to read the walkthrough tables. One row per step; read any column top-to-bottom to watch that lane evolve.  ROUTE = the app's own /api/* edge proxy (404 = flag off, 401 = no session, 200 = live).  ·  PLATFORM = the TWAI (the AI backend) behind that proxy.  ·  STORES = which of the three persistence stores is touched — Transcript (durable, platform-side, keyed by taskId), Snapshot (mirror/state/<runId>, a thin server asset = form answers + ids), Envelope (the browser's sessionStorage).  ·  RESUME? = is “Pick up where you left off?” shown here, and is that correct?

Use Case 4 — Active tab, mid-interview, opens /mirror
Before · on main today /mirror/report is the single entry chokepoint, and its resume gate checks the live tab envelope first — so a mid-interview tab that hits /mirror is sent straight back into the interview.
After · this change Entry now splits across /resume and /report, so the “continue this live tab” check moves into a one-line guard shared by both surfaces. The invariant is preserved: an actively running tab always continues its interview.

The conditions that don't fit a happy-path journey but decide whether the design is safe to ship. Each maps to a row in the decision table above.

Flag OFFproduction today release.mirror-return off (current prod). Row 1: first line of MirrorIndexPage ⇒ redirect("/mirror/start"). No cookie read, no resolver, no platform call. Byte-identical to prod
Zero behavior change ships to customers on merge. This is what makes the PR auto-mergeable.
Logged-outflag on No session cookie (flag on in preview/dev). Row 2: cookie-presence short-circuit ⇒ redirect("/mirror/start") before any platform work. Instant, matches today
The login wall lives on /start, so anonymous hits stay cheap.
Transient platform failurethe key rail Cookie present, but the resolver throws — platform 5xx, M2M mint fail, transient session-verify error, or the ~3500 ms timeout. That page re-probes; if the blip cleared it behaves normally, and if it is still failing it forwards to /mirror/report, whose retry UI takes over. Degrades to retry, never strands
A returning user with a real run is never wrongly dumped on /start during an outage; they reach an explicit “temporary hiccup, try again” state. Falling back to resume (not report) can never under-serve, since resume forwards to report anyway.
Misconfigured envno PLATFORM_URL Server env missing PLATFORM_URL. Row 3: resolver returns "start" immediately — can't read state with no platform. Safe default

4 · Accepted tradeoffs

Called out for sign-off, not buried.

1. One extra server read at entry. The resolver reads run state once server-side, and the destination page reads again to render. The reads are idempotent, and the entry read is capped by the ~3500 ms timeout, so worst-case entry latency is bounded and a slow platform can never hang the redirect. This is the price of being honest about the destination instead of guessing and bouncing.

2. Narrow brand-new-interview window. If a user is mid-interview in one tab and opens /mirror in a second tab within the first ~1.5 s — before the first snapshot is written — the resolver can't see the run yet and routes the second tab by server state alone. The #595 form-completion snapshot write shrinks this window to near-zero. Noted, not blocking.

3. The resume page re-probes. The entry decision is a hint, not a contract — /mirror/resume re-confirms with its own probe so a stale-by-milliseconds entry read self-corrects, and the shared in-flight-tab guard catches the live-tab case the server can't see.
