Clean-room design brief

Mirror — end-to-end flow, ground-up rethink

A brief for a fresh agent to design the complete Mirror user flow from first principles — every person, every branch, first touch to long-term return — as if no product existed yet.

Greenfield by design. It carries almost no current-state detail on purpose, so the agent proposes the right flow instead of reproducing an existing one. Click Copy brief as Markdown and paste it in as the agent's assignment.

01  What Mirror is (the minimum you need)

  • Mirror is an AI-guided interview. A person spends ~20–30 minutes in a guided conversation; at the end the system generates a personalized written deliverable for them — their Mirror (a reflective "AI-readiness" report). This deliverable is the entire payoff of the product.
  • The deliverable is saved to the person's account, so they can leave, return to it later, and accumulate more than one over time.
  • The interview and the deliverable generation run on a backend AI platform. The browser never calls it directly — an application server layer sits in between. Running the interview and generating/saving the deliverable take real wall-clock time and can be slow or fail.
  • Owning a saved deliverable implies a known identity (the person is signed in). Where sign-in sits in the flow is yours to design, but ownership requires identity.

02  The people who arrive

Design for every one of these. Each arrival must resolve to exactly one truthful destination.

1 · First-timer

Never visited; nothing started.

2 · Finisher, returning

Completed a Mirror before; has at least one saved deliverable.

3 · Abandoner, returning

Started an interview once but never finished; nothing was delivered.

4 · In-progress, right now

Partway through an interview this moment, maybe open in another tab.

5 · Power returner

Several Mirrors across multiple past sessions.

03  The core tension to resolve

Every time a person arrives, the app must decide — from who they are and what they've done — the single most truthful place to put them, in one clean step, with no misleading intermediate screens.

  • A deliverable a person deliberately finished is DONE: show it; never ask them to "resume" it.
  • The "pick up where you left off?" offer is right in exactly one situation: a genuinely abandoned, unfinished interview, seen on a later visit.
  • Slow or failed platform work must never strand the person, and never hide a deliverable they own.

04  What to design (answer all)

  1. Surfaces & routes. Enumerate the distinct surfaces a person can be on. Give each an honest URL — a URL must not claim to be something it routinely isn't.
  2. Entry resolution. At the entry point, how does the app decide where to send a person? Give a concrete model: inputs (identity, run state, history) → exactly one destination, for all five people. Make precedence explicit when rules could collide.
  3. Per-surface contract. For each surface: its single job, what it shows, what the person can do, and where they go next.
  4. Runtime vs platform actions. For each transition, two lanes. Runtime = app + its own server (navigation, identity/state read, render, calls to its own layer). Platform = the backend AI platform (start/continue interview, run inference, generate + save + retrieve the deliverable). Flag slow/async steps and define the waiting UI.
  5. Every branch, end to end. Walk each person's journey fully, including forks: resume → yes / no; interview ends naturally vs ended early; choosing among multiple Mirrors; re-opening while a session is live in another tab.
  6. Failure & edge handling. Signed-out, identity expired, platform slow, platform error, generation fails after the interview is over, conflicting state across tabs, stale link. Each lands on a real surface with a next step — never a dead end, never a false "you have nothing."
  7. Run lifecycle. Define the states one interview moves through (e.g. not-started → in-progress → finished → generating → delivered / failed; rename freely), and what each means for (a) routing and (b) whether "resume" is ever offered.

05  Deliverable format (produce all)

  • Route / URL map — table: route · surface · one-line job.
  • Entry-resolution model — first-match rule list or decision tree (inputs → one destination), precedence stated. Include a Mermaid flowchart of the entry decision.
  • Per-surface contracts — one short block per surface.
  • Runtime-vs-platform action table — two lanes, per transition; slow/async flagged.
  • End-to-end journey diagrams — one per person (Mermaid or ASCII), showing every branch.
  • Failure / edge matrix — condition → what the person sees → where they land.
  • Run-lifecycle diagram — the state machine for one interview.
  • Glossary — define every state and term you introduce.
  • Assumptions & open questions — what you assumed, what you'd confirm.

06  Ground rules & scope

Design ground rules

  • One clean landing per arrival — no visible bounce through intermediate URLs.
  • Honest URLs — the address reflects what the surface actually is.
  • Never hide a deliverable the person owns; never offer to resume something finished.
  • Graceful degradation — every failure resolves to a real surface with a next action.
  • Identity-gated where ownership matters, but keep first-touch friction low.
  • Design at the product / UX + flow level — no assumed framework, language, storage model, or file layout.

Out of scope

Visual design, branding, final copy. Tech stack, data schema, code structure. Any existing implementation — ignore it entirely; this is greenfield.

07  Posture

Design as if the person on the other end paid for the deliverable, and as if their first arrival and every return are the whole product. Optimize for one thing: they always land exactly where they should, the moment they arrive — finished work always within reach, unfinished work offered back only when it truly is unfinished.
