M2M migration — what changes, in plain words

Three ways to build the stakeholder interview, side by side. Today (held together with tape), Parity-Plus (same product, clean internals), and Next-Gen (a small team of cooperating AIs). Written for non-technical reviewers: defaults are plain English; the engineering bits sit behind the toggle on the right.

Section B

Phase by phase

Same seven phases, more detail. Each phase shows three columns of “who does what” for Today, Parity-Plus, and Next-Gen. Read across a row to see the difference between shapes.

Phase 1 of 7

Invited

A stakeholder receives a link from us and clicks it. The app loads in their browser and asks them to sign in. This phase is identical across all three shapes — the change story starts in Phase 2.

TodayRube Goldberg

Actor Action & why
Stakeholder Clicks the invitation link. Wants to share their input.
App Loads in the browser, asks for login. Needs to know who's coming in.
External Login service confirms the email belongs to the right organisation. Gatekeeper.
App Now in a trusted state, shows the brief form. Need the stakeholder's context before the AI starts.
Under the hood
  • Auth lives in @stytch/nextjs/b2b, fully browser-side. Session JWT lands in stytch_session_jwt cookie.
  • No server-side verification of that cookie today — the bapps server is a dumb forwarder.

Parity-PlusSame product, clean internals

Same as Today on the surface.

Under the hood
  • Browser-side login is unchanged, but the bapps server now verifies the session cookie server-side via stytch.sessions.authenticate before any AI call.
  • The verified { org_id, member_id } becomes the trust anchor for everything downstream.

Next-GenFirst-principles, agentic

Same as Parity-Plus.

Under the hood
  • Trust anchor unchanged. All the new stuff happens downstream of login.

Phase 2 of 7

Set up

The interview brief — the form data the stakeholder just filled in plus the scoping info from us — has to reach the AI before the conversation starts. This is where the most invisible work happens, and it's the biggest difference between the three shapes.

TodayRube Goldberg

Actor Action & why
Stakeholder Fills in the brief form. Sets context before talking to the AI.
App Stuffs the brief into a fake user message wrapped in hidden markup. Only way to get instructions to the AI today.
Bapps server Passes the message through to the platform unchanged. Dumb forwarder.
TWAI Platform Bounces the fake message off a “transcriber” AI that copies the text back verbatim, then stores it. Only way to plant setup instructions in the right slot.
Under the hood
  • <hidden>...</hidden> wrappers added in apps/web/lib/prompts/assemble.ts.
  • context-asset-transcriber step + a separate chatStream round-trip in apps/web/lib/use-prepare-task.ts.
  • Fidelity check echoed text against the intended prompt — ~1-3% drift on long prompts.

Parity-PlusSame product, clean internals

Actor Action & why
Stakeholder Fills in the brief form. Same purpose as Today.
App Sends the brief to the bapps server. No hidden tags, no AI round-trip. Stops faking it.
Bapps server Assembles real setup instructions and hands them to the platform when the AI step starts. Sets the AI up properly in one go.
TWAI Platform Uses those instructions directly as the AI's setup. One round-trip disappears.
Under the hood
  • promptOverrides.system on the first __AISTART__ request (layer-4 system-prompt override).
  • New /api/m2m-chat route on bapps server; M2M token via Stytch client-credentials; delegation via X-On-Behalf-Of: <org>:<member>.
  • Server-side stytch.sessions.authenticate derives the X-On-Behalf-Of values — never trust the client.

Next-GenFirst-principles, agentic

Actor Action & why
Stakeholder Fills in the brief form (or speaks it). Same purpose.
Bapps server Assembles three separate briefs — one each for Interviewer, Observer, and Synthesis. Each AI has a narrow job.
Bapps server Loads past discoveries about this user from long-term memory. Pick up where we left off.
TWAI Platform Starts three AIs, each with its own brief. Real specialisation.
Under the hood
  • Three parallel taskStepChat sessions (Interviewer, Observer, Synthesis), each with its own promptOverrides.system.
  • GET /api/context-assets at scope: user to read prior discoveries.
  • Bapps server orchestrates fan-out + merge — platform has no “agent” primitive.

Phase 3 of 7

Opener

The AI greets the stakeholder and asks the first question. Same outcome in all three shapes; the difference is what just happened in Phase 2 and what's already running in the background.

TodayRube Goldberg

Actor Action & why
TWAI Platform Triggers the AI's first turn using the setup just planted. Standard kickoff.
AI Says hello, asks first question. Opens the conversation.
App Renders the question on screen. Stakeholder sees it.
Under the hood
  • __AISTART__ AIStart trigger on the interview step.
  • Roughly 2-5 seconds slower than necessary because Phase 2 waited on the transcriber round-trip.

Parity-PlusSame product, clean internals

Actor Action & why
TWAI Platform Triggers the AI's first turn using the new direct setup. No round-trip held up the start.
AI Says hello, asks first question. Same outcome, snappier.
App Renders the question on screen. Stakeholder sees it.
Under the hood
  • Same __AISTART__ trigger. Setup is baked in via promptOverrides.system on the first call — no separate request — so the opener fires straight after Phase 1.

Next-GenFirst-principles, agentic

Actor Action & why
TWAI Platform Triggers the Interviewer AI's first turn. Warm-up.
AI Interviewer says hello, asks first question. Conversation starts.
Observer AI Watches the question land in the background. Scoring coverage from turn 1.
App Renders the question; optional voice playback. Audio if requested.
Under the hood
  • Three __AISTART__ triggers in parallel.
  • Interviewer's response streams to the user; Observer and Synthesis run silently against the same transcript.
  • Voice token from /api/voice/token (Deepgram) if the stakeholder chose voice.

Phase 4 of 7

Conversation

The back-and-forth interview loop. Most of the time is spent here and most of the value is created here. Today and Parity-Plus run the same loop; Next-Gen is where the shape genuinely diverges.

TodayRube Goldberg

Actor Action & why
Stakeholder Answers the question. Real input.
App Sends the answer as a chat message. Standard turn.
Bapps server Passes it through to the platform. Dumb forwarder.
AI Reads the answer, asks the next question. Tries to be warm, drive coverage, and remember everything at once.
App Renders the follow-up; loops back. Standard chat loop.
Under the hood
  • chatStream on the interview step. Each turn = one HTTP request from the browser.

Parity-PlusSame product, clean internals

Same loop as Today. Parity-Plus changes setup (Phase 2) and wrap-up (Phase 5) plumbing — not the conversation itself.

Under the hood
  • Same chatStream path, but the new { role, content } body shape (TWAI 1.8.0+).
  • No transcriber leg running anywhere.

Next-GenFirst-principles, agentic

Actor Action & why
Stakeholder Answers (text or voice). Real input.
Bapps server Forks the answer to all three AIs. All three see the same transcript.
AI Interviewer asks the next question and quietly emits structured insight fragments alongside the visible reply. Doubles as data collection.
Observer AI If a topic is thin, quietly nudges the Interviewer (“ask for a concrete example here”). Mid-flight quality control.
Synthesis AI Drafts insight fragments as the interview runs. Report is 80% written before wrap-up.
App Renders the follow-up; may show a live “insights forming” panel. Optional transparency.
Under the hood
  • Three parallel chatStream sessions on three task steps.
  • Observer's nudges = role: developer append-without-submit (submitToLlm: false) on the Interviewer's session.
  • Synthesis emits typed fragments in the data field of its chat response; bapps accumulates them in a typed insight store.
  • Pivoting the Interviewer's persona mid-conversation needs a real promptOverrides swap — platform doesn't allow that today; approximated with a strong role: system append.

Phase 5 of 7

Wrap-up

The AI signals it has enough and closes politely. In all three shapes the outcome looks the same to the stakeholder; the mechanics differ a lot.

TodayRube Goldberg

Actor Action & why
AI Decides it has enough, says so. Coverage check is implicit in the prompt.
App Sends a hidden “generate report” trigger disguised as a user message. Only way to flip the AI into report mode today.
Bapps server Passes the hidden trigger through. Dumb forwarder.
AI Receives the hidden instruction and switches to report mode. Hack works; trigger crosses the wire.
Under the hood
  • <hidden>/generate-report</hidden> smuggled in a role: user message from the browser.
  • Visible to anyone with browser DevTools open.

Parity-PlusSame product, clean internals

Actor Action & why
AI Decides it has enough, says so. Same coverage logic.
Bapps server Sends the “switch to report mode” instruction as a real system message (not a fake user message). Clean.
AI Receives the system message and switches. Trigger is server-side and invisible to the browser.
Under the hood
  • role: 'system' append with submitToLlm: true — or promptOverrides.system on a separate report step's __AISTART__.
  • Server-side only; no smuggling across the wire.

Next-GenFirst-principles, agentic

Actor Action & why
Observer AI Decides coverage is good enough; signals wrap-up. Coverage is its job now.
Bapps server Tells the Interviewer to close warmly. Hand-off.
AI Interviewer closes politely. Good UX.
Synthesis AI Final assembly pass on the insights already drafted. Already 80% done.
Under the hood
  • Observer emits a role: developer “wrap up now” message to the Interviewer.
  • Synthesis's accumulated data fragments are the source of truth for the report.

Parity-PlusSame product, clean internals

Section C

Where the platform falls short of world-class

To build Next-Gen fully, we'd ask the platform team for a few things. None of these block Parity-Plus; they bound how far Next-Gen can go.

  1. The AI can't directly request a tool or trigger an external action mid-conversation. Every “AI asks to upload a file / book a meeting / call out to something” has to be faked with text patterns we then parse. a tools field on chat input, a tool_calls channel on the response, and a role: tool response type.
  2. An AI can't change its own instructions mid-interview. The system-prompt override only locks in at the start. The Observer AI can nudge with system / developer messages but can't swap the Interviewer's persona. relax promptOverrides timing from “first request of session” to “any request before submitToLlm=true”.
  3. The platform has no built-in notion of a second AI watching the same conversation. We have to wire fan-out, merge, and ordering ourselves. a first-class “agent” or “sub-task” concept on the task model, with shared state and ordering primitives.
  4. Structured insight extraction works, but the platform doesn't enforce the shape. The data field is freeform; bapps validates on its side. optional JSON-schema validation on data per step.
  5. Voice pacing / interruption / barge-in is fully in the browser. The platform mints a transcription token (Deepgram); everything else — turn-taking, mid-utterance pauses — is client-side. hooks for prosodic / pacing metadata in the chat protocol (lower priority).
