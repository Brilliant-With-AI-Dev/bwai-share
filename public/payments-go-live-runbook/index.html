Stripe + Resend — go-live runbook

The code for self-serve Individual Mirror payments is built, merged, and deployed. Nothing charges a real card yet because the live keys aren't set. This runbook is the safe path from "deployed" to "taking real money": test everything on a preview first, prove a full purchase and a full refund work end-to-end, and only then flip on live keys in production.

Code shipped — keys not yet live Stripe Checkout + webhook Resend transactional email Production env writes are Matt-only
Green = Claude can do it (code, config files, preview verification) Red = Matt only (production keys, secrets, going live)

00  The one rule

⚠️
Test mode first. Always. Stripe has two completely separate worlds — test mode (fake cards, no money) and live mode (real cards, real money). They have different keys, different webhook endpoints, and different product prices. We wire up and prove the entire flow in test mode on a preview deployment. Live keys go into production last, only after a test-mode purchase and a test-mode refund have both worked end-to-end.
🔒
Who flips the switches. Claude writes the code and config files and runs the preview verification. But anything that touches production environment variables, secrets, or the live Stripe toggle is Matt's to do — those are customer-facing and irreversible-by-surprise, so they never get done automatically. Every such step below is marked Matt only.

01  What you're turning on

Three pieces, already built and on main:

  • Stripe Checkout → access grant. The buyer pays on Stripe's hosted checkout. Stripe then calls our webhook, and the webhook grants them access to Mirror Pro. (Built in #610.)
  • Refund / dispute → access revoke. If a payment is refunded or disputed, the same webhook automatically revokes access, fail-closed. (Built in #613.)
  • Transactional email. Sign-in links and the purchase confirmation are delivered through Resend. While its keys are blank, email is dormant and simply does nothing — no crash. (Built in #458.)
🔌
The single webhook endpoint Stripe calls is /api/stripe/webhook. It must be subscribed to exactly three events: checkout.session.completed (grant), charge.refunded and charge.dispute.created (revoke). Configuring that endpoint is Step 3 below.

02  The environment variables

Six variables across two services. The Preview column is what Claude/Matt set on a preview deployment with test-mode values; the Production column is the live value Matt sets last. Every name below is the exact string the code reads — see apps/web/.env.example.

Stripe

Variable What it is & where it comes from Scope
STRIPE_SECRET_KEY The server API key. Stripe Dashboard → Developers → API keys. Test value starts sk_test_…, live value sk_live_…. Preview: test Prod: live
STRIPE_WEBHOOK_SECRET Signing secret that proves a webhook call really came from Stripe. Created per endpoint when you add the webhook (Step 3), shown as whsec_…. Test and live endpoints each have their own secret. Preview: test Prod: live
STRIPE_PRICE_INDIVIDUAL_MIRROR The price ID for the Individual Mirror product. Stripe Dashboard → Product catalog → the product → its price, copy the price_… ID. The test-mode and live-mode catalogs are separate, so this ID differs between them. Preview: test Prod: live
🧾
Sales tax (#615). Stripe Tax is configured in the Stripe Dashboard (Settings → Tax), not via an env var. Turn on automatic tax in test mode and confirm the checkout shows tax before enabling it in live mode. .env.example carries a note pointing here.

Resend (email)

Variable What it is & where it comes from Scope
RESEND_API_KEY Resend API key. Resend dashboard → API Keys, shown as re_…. Leave blank to keep email dormant. Preview Prod
EMAIL_FROM The "from" address, e.g. Mirror <noreply@mail.brilliantwith.ai>. The domain must be verified in Resend first (DNS records — see Step 2). Until verified, sends are rejected. Preview Prod
EMAIL_REPLY_TO A monitored reply-to address, e.g. support@brilliantwith.ai. Optional but recommended so replies reach a human. Preview Prod
✅
All three Resend variables blank = email stays dormant and the app runs fine. That means you can stand up and test the Stripe flow first, and add email afterward, without breaking anything.

03  The go-live sequence

Run these in order. Steps 1, 3, and 5 are Matt's (they set secrets or go live); the rest Claude can drive.

1

Create test-mode keys & the test product Matt only

In the Stripe Dashboard, with the toggle set to Test mode:

  • Copy the test STRIPE_SECRET_KEY (sk_test_…).
  • Create the Individual Mirror product + price, copy its price_… for STRIPE_PRICE_INDIVIDUAL_MIRROR.
  • Set both on the Preview environment in Vercel (Settings → Environment Variables). The webhook secret comes in Step 3.
2

Verify the email domain in Resend Matt only

In Resend, add and verify the sending domain (it lists the DNS records to add at your DNS provider). Once verified, set RESEND_API_KEY, EMAIL_FROM, and EMAIL_REPLY_TO on Preview. Email can also be left dormant for the first Stripe test and added here afterward.

3

Add the test webhook endpoint Matt only

Still in Test mode: Stripe Dashboard → Developers → Webhooks → Add endpoint.

  • URL = https://<your-preview-deployment>/api/stripe/webhook
  • Subscribe to the three events: checkout.session.completed, charge.refunded, charge.dispute.created.
  • Copy the endpoint's signing secret (whsec_…) into STRIPE_WEBHOOK_SECRET on Preview, then redeploy so it picks up the value.
4

Prove it end-to-end in test mode Claude can drive

On the preview deployment, with Stripe in test mode:

  • Purchase: buy a Mirror with Stripe's test card 4242 4242 4242 4242. Confirm the buyer lands on the claim page, claims the account, and ends up with Pro access.
  • Refund: refund that test payment from the Stripe Dashboard. Confirm access is revoked automatically (the charge.refunded path).
  • Email: if Resend is wired, confirm the sign-in link and confirmation email actually arrive.
  • Check the webhook's delivery log in Stripe shows 200 responses, not retries.
5

Go live — production keys last Matt only

Only after Step 4 fully passes. Flip Stripe to Live mode and repeat the key collection for real:

  • Live STRIPE_SECRET_KEY (sk_live_…) and the live STRIPE_PRICE_INDIVIDUAL_MIRROR → Production in Vercel.
  • Add a second, live-mode webhook endpoint pointing at the production URL (https://brilliant-apps.vercel.app/api/stripe/webhook), same three events; put its whsec_… into the Production STRIPE_WEBHOOK_SECRET.
  • Set the live Resend values on Production.
  • Enable automatic tax in live mode (if using #615).
  • Redeploy production so it reads the new values.

04  First-live smoke check

Immediately after going live, do one real low-value purchase yourself and confirm each box, then refund it:

  • Checkout completes and charges the real card.
  • Buyer reaches the claim page and can claim via Google or magic link.
  • Pro access is actually granted after claiming.
  • Confirmation email arrives from the live EMAIL_FROM.
  • Stripe webhook log shows 200 for checkout.session.completed.
  • Refund the test purchase — confirm access is revoked.
  • Tax line is correct on the receipt (if #615 is on).

05  If something's wrong

↩️
Fastest kill switch: in Stripe, disable the live webhook endpoint (Developers → Webhooks). Grants stop immediately because the grant fires from that webhook. Existing customers keep their access; no new ones are minted. Fix, re-enable, and Stripe will replay the missed events. To stop new purchases too, take down the checkout entry point or remove the live secret key — but disabling the webhook is the surgical first move.
🔁
The webhook is idempotent — it remembers each Stripe event ID, so a replay or duplicate delivery won't double-grant or double-revoke. Safe to let Stripe retry.
